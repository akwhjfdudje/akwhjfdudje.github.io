Recursive Descent

Hello. This post is about recursive descent.

Parsing

We will start with some background. Compilers, for a long time now, have been made with a particular design in mind. Said design involves the stages of lexing, parsing, analysis, and code generation.

Note some things at this point:

  • This is not the only way to read programming languages.
  • The stage of "analysis" nowadays contains several "substages", which are not important for now.
  • "Code" in "code generation" here refers to the direct assembly that a compiled language turns into. This is, of course, not the only output of a compiler. There are certainly several more "code"s that a compiler may output (that are not pure assembly).

The stage of lexing will be left as a black box for now. All that is important is that lexing turns a sequence of words, symbols, etc. into a set of "tokens".

The stage of parsing takes the set of "tokens" from lexing, and attempts to give "meaning" to the "tokens". Which method is chosen depends on the design of the parsing stage, the needs of the language developer, opinions, etc.

The topic of this post, recursive descent, is one of the aforementioned techniques.

Theory

Compiler design relies on a formal theory of languages. The significant aspect of this theory here is the concept of a grammar. A grammar consists of:

  • Variables: symbols that can be expanded
  • Terminals: symbols that cannot be expanded
  • Rules: ways to expand variables into variables/terminals
  • Start: symbol that begins evaluation

Here is an example: 

  Expr -> Expr + Term | Term
  Term -> Term * Factor | Factor
  Factor -> (Expr) | id

Here, the variables are Expr, Term, and Factor, the rules are the lines, the terminal being "id", and the start Expr.

The Algorithm

With some theory provided, we cover the algorithm itself.

The basic idea is that, in code, each aspect of a formal grammar described above becomes a part of the code. As in, the variables map to functions, terminals map to tokens, and rules map to what goes on in the functions themselves.

As should be clear, this makes recursive descent a very easy to implement algorithm, as it closely matches the grammar with the code.

However, this algorithm, like most, is not without its limitations. We cover one of the limitations in this post.

Left Recursion

Recall the grammar from above. Notice the following rule:

Expr -> Expr + Term | Term

In recursive descent, this will be written in code as something like the following: 


  def parseExpr():
      parseExpr()
      match('+')
      parseTerm()

Notice how the function does not ever stop calling itself. This is exactly what one of the biggest issues with recursive descent is.

Conclusion

There is certainly more to recursive descent than this post. However, I hope that this serves as a short and simple introduction to recursive descent and formal language theory.
